How MemCP went from corrupting itself to lossless cross-machine sync

Two machines share one memory store through Google Drive. Getting that right took four moves — from "both machines scribble on the same file" (which literally corrupted the database, twice) to a design where concurrent writes can't lose a byte. This is the whole arc, in plain terms.

00The four eras, at a glance

Each era fixed the failure the previous one exposed. The first two are about resource contention (two writers colliding on shared storage). The last two are about not losing data once contention is survivable.

original setup

01Where it started: two machines, one file, corruption

The original design was the obvious one: put graph.db in the Google Drive folder, point both machines' MEMCP_DATA_DIR at it, and let Drive sync the file. One canonical store, no divergence. The problem is that a synced cloud folder is not a real shared filesystem.

The original contention bug: corruption, not lost rows. It hit twice and took every operation down with it.

what actually happened

Two corruption incidents

On 2026-05-06 and again 2026-06-01, graph.db threw "database disk image is malformed" on every operation — even ping. Integrity check showed a torn b-tree page. Recovery via SQLite's .recover got back ~99.9% (lost 2 low-value rows the second time). The corrupt originals still sit in Drive as graph.db.malformed-2026….

why it was inevitable

Three compounding hazards

  • FUSE sync tears pages. Drive copies the file in the background; a write in progress becomes a half-synced file on the other side.
  • Locks don't cross Drive. SQLite's advisory locks assume a real FS; on Drive they're meaningless.
  • Even reads write. A recall bumps an access counter — so just reading on both machines was concurrent writing.

02Stop the contention: lock the writes, then take the DB off Drive

This came in two moves. The first made concurrent writes survivable; the second made the dangerous sharing impossible.

Move A — a write-grained lock + DELETE journal

The instinct "give each machine a turn for the whole session" was wrong — it'd force the second window read-only and defeat the point of running two sessions at once. Instead the lock is write-grained: grab it right before a save, release right after. Reads never lock. Concurrent sessions coexist; only the millisecond-long commits queue.

rock-solid tier

Local flock on a lockfile kept on local disk (never the synced mount, where flock is meaningless). Serializes every writer process on one machine instantly and reliably.

best-effort tier

Cross-machine lease — a .writer.lock next to the DB on the synced mount holding {host, pid, heartbeat}. Bounded by Drive's latency, it catches "the other machine is actively writing"; a crashed holder's stale lease is reclaimed by TTL.

Plus a switch to DELETE journal mode — SQLite writes the main file directly instead of leaving -wal/-shm sidecars that Drive would sync separately and tear. The lock is also fail-open: any error in the locking machinery logs and proceeds rather than ever blocking or dropping a save.

Move B — the live DB goes local; only a snapshot rides Drive

A lock makes a shared file safer, but two machines writing one file over Drive is still fragile, and Drive sync-lag still leaves a "B overwrites A before A's last write synced down" window. The clean cure is to stop sharing the live file at all:

After Move B, the only thing crossing Drive is a snapshot file — a publish/subscribe copy — not the database the engine reads and writes. That kills corruption entirely. But it opens the door to a second, gentler problem: how do two snapshots merge without losing rows? That's eras 3 and 4.

03The key idea behind no-loss merge

With the live DB local, the snapshot layer still shipped a data-loss bug: push() overwrote the whole Drive snapshot with the local file and minted a higher version number — no row-level merge. A stale machine (even doing a read) could republish its older DB over the snapshot and erase the other machine's new insights. The fix follows from one rule:

Local-DB monotonicity

A pull only ever adds rows to the local DB. It never deletes a local row to match the snapshot.

Add "every session pulls, then pushes," and the guarantee is automatic: if a row ever drops out of the shared snapshot during a race, it still lives on the machine that created it — and re-publishes on that machine's next push. Nothing is gone; it just re-propagates a cycle later.

This is why the heavy machinery in era 4 (immutable files, pointers, hashes) is a robustness layer, not a correctness requirement. Correctness rests entirely on the rule above. Even a badly raced publish can't lose data — it can only delay convergence.

One honest trade-off it buys: because a pull keeps the local copy of a row it already has, a re-tag or importance change on one machine doesn't propagate to the other. Both copies of the content always survive — they can just disagree on metadata. Accepted: those are internal classification fields, not your memories.

04The no-loss merge, concretely

Pull = the only place a merge happens (additive union)

  • INSERT OR IGNORE pulls in any row we don't have; existing ids keep our copy.
  • The tombstones (delete log) union in too, then apply as a deny-set — so "present in snapshot and tombstoned" resolves to deleted on every machine, whatever the arrival order.
  • A foreign-key check runs before commit; a bad merge rolls back instead of corrupting the DB.

Push = publish a superset, never overwrite

1
Catch up first — if the snapshot moved ahead, fold it in so local is a true superset.
2
Write a globally-unique, immutable blob — graph.snapshot.<gen>.<host>.<rand>.db. Two machines can never pick the same name, so neither can clobber the other.
3
Flip a tiny pointer — graph.snapshot.ptr.json names the current blob + its content hash. Readers follow the pointer.
4
Self-verify on read — generation + hash live both inside the blob and in the pointer; a torn or half-synced blob is rejected, not adopted.

Append-only blobs + a one-line pointer. A publish race can no longer destroy a write — it just leaves an extra file.

"Contention" = more than one writer touching shared state at once. Across the four eras, four distinct collisions were each neutralized. None can now lose data or corrupt the store.

Why there's no global lock for the cross-machine case

A Drive-synced lockfile rides the very sync latency it's trying to police, and a crashed holder strands the system — so the cross-machine lease is deliberately best-effort only. The real guarantee comes from making overwrites impossible (immutable blobs) and the local DB monotonic. You don't prevent concurrent writes — you let them both happen and merge. That's the elegant part, and it's what Jeff's "no bespoke locking scheme" constraint pushed toward.

The throughline: contention is made safe by construction, not prevented by coordination. A lost race costs one extra merge cycle — never a memory, and never the database.

Backstops, for when a race still slips through

  • Tombstones — every delete writes an id+timestamp record, kept forever, that travels in the union and re-kills the row everywhere (so a delete can't be silently resurrected by a stale peer).
  • Re-push outbox — a machine re-publishes any of its own rows a peer hasn't echoed back yet, so a row is never stuck as a single copy.
  • Generation-aware GC — old blobs are reaped only once every known machine has merged past them.
  • Format gate — an older binary refuses to read or overwrite a newer-format snapshot, which is what makes a one-machine-at-a-time upgrade safe.
  • ingest_seq reindex — merged-in foreign rows (which carry their origin's older date) are still indexed and searchable in the same session, instead of looking lost.
